# Login Token

Explains `login-token.schema.json` (version `1.0.0`, the authoritative definition —
this file explains, it does not define, per `shared/README.md`'s format rule).

## Scope: IA-1 is infrastructure only

This contract, and `apps-script/FoundationLoginTokens.gs`'s implementation of it, are
IA-1's entire deliverable — the first of two independent Identity & Access batches
this milestone was explicitly split into. IA-1 delivers token generation, hashing,
expiration, and single-use enforcement only. It does **not** issue a Session
(`shared/schemas/session.md`), does **not** send an email, does **not** expose a Web
App route, and does **not** implement any authentication UI — `foundationConsumeLoginToken_()`
returns a resolved `patient_id` and nothing more; a future batch (IA-2) is what calls
`foundationIssueSessionToken_()` with that `patient_id` to actually log someone in.

## Why only the hash is ever stored

docs/29-PHASE-2A-TECHNICAL-PLAN.md §3 is explicit: "generates a random 256-bit token,
stores **only its hash** in `LoginTokens` (never the raw token)." A stolen or leaked
`LoginTokens` export must never be usable to authenticate as anyone — the same
principle a password-reset-token or API-key system applies. `token_hash` is a plain
SHA-256 digest of the raw token (`apps-script/FoundationLoginTokens.gs`'s
`foundationHashLoginToken_()`, via `Utilities.computeDigest(Utilities.DigestAlgorithm.SHA_256, ...)`)
— not HMAC. This is a deliberate choice, not an oversight: HMAC exists to authenticate
a *message* using a shared secret (exactly what `FoundationSession.gs` needs, since a
session payload is attacker-supplied and must be proven to come from this server).
A login token is different — it is *generated by this server*, high-entropy, and
random; hashing it is only ever "store a fingerprint of a secret we already
control," which a plain cryptographic hash already does correctly. Adding an HMAC
secret here would add a second secret-management surface (a new Script Property) for
no real security benefit against a token with well over 256 bits of entropy — the same
"avoid over-abstraction at pilot scale" reasoning ADR-006 already applies elsewhere in
Foundation.

## Where the raw token's entropy comes from

`foundationGenerateRawLoginToken_()` concatenates three `Utilities.getUuid()` calls
(hyphens stripped) — the same native primitive `FoundationUtils.gs`'s
`generateFoundationId_()` already uses, and the only cryptographically-backed
randomness source Apps Script exposes without a new API surface. Each RFC 4122 v4 UUID
contributes ~122 bits of real randomness (8 of its 128 bits are fixed version/variant
bits); three concatenated comfortably clears docs/29 §3's 256-bit target with margin,
without introducing a new randomness primitive this codebase has never used before.

## The `used_at` sentinel value — a new pattern for this repository

Every other Foundation schema (`patient-identity`, `session`) has exclusively
always-set fields. `LoginTokens` is the first entity with a field that is legitimately
absent at creation and filled in exactly once, later: `used_at`. `FoundationDataStore.gs`'s
existing row/object conversion already writes an empty string for any missing field
(`foundationDsObjectToRow_()`'s documented behavior) — rather than invent a new
null-handling convention, `used_at` reuses that existing convention directly: `''`
means "not yet used," any non-empty value is the ISO 8601 consumption instant. The
schema's `used_at` property is therefore typed `"string"` only, deliberately without
`format: "date-time"` — that format check would incorrectly reject the legitimate,
intentional empty-string sentinel. `login-token.schema.json` still requires the key
be present (`required` includes `used_at`) — every record read back from
`FoundationDataStore.gs` always has all five columns, never a genuinely missing key.

## Fields at a glance

| Field | Set at creation? | Mutable? |
|---|---|---|
| `token_hash` | Yes | No — permanent, the row's effective identity |
| `patient_id` | Yes | No |
| `issued_at` | Yes | No |
| `expires_at` | Yes | No |
| `used_at` | Yes (as `''`) | Once — `''` → consumption timestamp, never again (single-use) |

## Rejection is deliberately generic

`foundationConsumeLoginToken_()` collapses every failure mode — not found, expired,
already used, malformed input — into one outward error code,
`FOUNDATION_LOGIN_TOKEN_INVALID`, with one friendly, non-technical message. The
*specific* reason is still recorded (in a `FoundationAudit.gs` `login_token_rejected`
event's `detail` field, and in the internal `foundationEvaluateLoginTokenRecord_()`
result used for testing), but never returned to the caller — the same anti-enumeration,
security-over-convenience reasoning (ADR-010) docs/29 §3 already applies to the
request-link step ("The response is identical whether or not the email matched a
patient"), extended here to the consume step.

## Versioning

Version `1.0.0`. Any field addition, removal, or type change requires a new version
here first, then a subsequent update to `apps-script/FoundationLoginTokens.gs` — never
the reverse, per `shared/README.md`.
